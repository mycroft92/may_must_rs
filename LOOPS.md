# Loop Invariant Checking

This document describes how `check_loop_invariant_verbose` in
`src/may_must_analysis/loops.rs` checks a candidate invariant, and the
soundness status of each step.

---

## Three-Stage Check

### 1. Initiation

```
forward_reach_at_header(cfg, header, inner) ∧ ¬I   →   SMT infeasible
```

`forward_reach_at_header` runs **SP (strongest postcondition)** forward from
the function entry with all back edges excluded. At join points the formulas
from incoming paths are OR-ed and the concrete store facts are **intersected**
(only facts that hold on every incoming path survive). The result is augmented
with those surviving facts as `select(region, k) = value` equations, giving the
solver concrete numbers (e.g. `j == 0`, `SIZE == 1`) that would otherwise be
lost behind uninterpreted `select` terms.

Because SP with excluded back edges **over-approximates** the actual first-entry
states, infeasibility of `reach_h ∧ ¬I` soundly implies I holds on every real
first-entry state: the over-approximation contains all real states, so if no
state in it violates I, no real state does either.

### 2. Inductiveness

```
I  ⊢  WP_hoare(body, I)
```

`backward_states(…, inductive_assume: true)` propagates the requirement `I`
backward from the latch (via the back edge, then through the body) to the
header. Node effects use `wp_inductive` instead of the standard `wp`:

```rust
// wp_one (standard violation semantics):
TransferEffect::Assume(c) => Formula::and(c.clone(), post.clone()),

// wp_one_inductive (Hoare-style, used for inductiveness only):
TransferEffect::Assume(c) => Formula::implies(c.clone(), post.clone()),
```

**Why Hoare-style for inductiveness is correct.** If the loop body contains
`Assume(v ≥ 0)` where `v` is the return value of a `nondet_int()` call, `v` is
a fresh variable for each iteration — no invariant over program state can bound
it. Standard WP adds `v ≥ 0` as a conjunct, making
`oracle.implies(I, v ≥ 0 ∧ …)` unprovable for every candidate. Hoare-style
produces `v ≥ 0 → WP_rest(body, I)`, equivalent to
`(I ∧ v ≥ 0) ⊢ WP_rest(body, I)` — inductiveness is only required for
executions that actually pass through the assume. This is the **semantically
correct** criterion: standard WP incorrectly demands that I imply the assume's
condition, which is an unnecessary restriction.

**This is not a soundness trade-off.** The Hoare variant never accepts a
non-inductive invariant; it only avoids spurious rejections caused by fresh
call-return variables.

One small inconsistency: `edge_source_requirement` for the back edge uses
**standard** `wp` for the latch node (not Hoare). Body nodes thereafter use
Hoare-style. In practice the latch is an increment node with no `Assume`
effects, so this gap does not cause problems. If a latch node ever had an
`Assume` on a fresh nondet-return variable, the affected candidate would be
spuriously rejected (conservative, not unsound).

### 3. Exit Closure (conditional)

For each exit edge whose target has a non-trivial assertion postcondition:

```
I  ∧  backward_propagation_of_violation_to_header   →   SMT infeasible
```

The violation condition is propagated backward (using standard `wp`,
`inductive_assume: false`) from the assertion through the exit edge and the
loop body to the header. If the combined formula is infeasible, the invariant
blocks all violations through that exit.

**Why exit closure can fail for loops with nondet calls.** `nondet_int()` in
the loop body is a Call with `CallMemoryEffect::HavocMemory`. The standard `wp`
for `Call { .. }` is transparent (`post.clone()`), and the accompanying
`HavocRegions` effect existentially quantifies away all memory terms. After
backward propagation through this havocing, region-indexed terms like
`select(menor_region, 0)` are dropped, leaving only scalar constraints
(e.g. `j ≥ 0`). The combined formula `I ∧ (j ≥ 0)` is trivially satisfiable,
so exit closure always fails for such loops regardless of how strong I is.

## Entry-Safety Candidates: All Three Checks Required

Entry-safety candidates (generated by `entry_safety_candidates` in `loops.rs`)
have the form `counter == init_value || safety_condition`. They go through the
full three-stage check — initiation, inductiveness, and exit closure — like all
other candidate types. The real `assertion_postconditions` map is passed to
`check_loop_invariant_verbose`; no check is skipped.

### Soundness constraints on candidate generation

Two additional invariants protect against degenerate candidates:

**Concrete-integer filter.** Only preheader store facts whose stored value is a
compile-time integer constant are used to build `counter_eq`. Facts from
`nondet_int()` calls (or other unresolved pointer arguments) produce
variable-valued facts such as `(menor_stack, 0) → retval_N`. Using these would
generate candidates like `select(menor_stack, 0) == retval_N || safety`.
Such a candidate trivially satisfies initiation — `forward_reach_at_header`
already encodes `select(menor_stack, 0) == retval_N` as a store-fact equation,
so the violation `reach_h ∧ ¬candidate` is a contradiction. It also passes
inductiveness vacuously. Because the invariant is not genuinely inductive,
injecting it into `reach` can make `reach ∧ state` infeasible at entry even
for unsafe programs — a false Verified. (Observed in `array-2`, fixed v0.10.1.)

**Whole-formula negation.** The safety formula is `negate_comparison(violation)`
or `Formula::not(violation)` applied to the whole exit violation, not to each
conjunct. Per-conjunct negation pulls in type-bound assumptions as conjuncts
and negates them to spurious safety atoms (`j < 0`). These form semantically
vacuous disjuncts that inflate the search space without helping any candidate
discharge the assertion.

### Historical note: Phase-B (removed v0.11.0)

In v0.9.0–v0.10.x entry-safety candidates were accepted without exit closure
("Phase-B"). The argument was that `run_backward` would discharge the assertion
anyway. This was found to be unsound: `run_backward` blocks back edges and
injects invariants only at loop headers, so it cannot reason about state after
one or more iterations when the invariant itself does not rule out the violation
path. Phase-B was removed in v0.11.0. All three checks are now required for
every candidate that reaches `run_backward`.

## Soundness Summary

| Check / approximation | Status |
|---|---|
| Initiation via forward over-approximation | Sound |
| Inductiveness via Hoare-style WP for `Assume` | Sound (correct semantics) |
| Store-fact intersection at join points | Sound (conservative) |
| All three checks required for every `VerifiedLoopInvariant` | Sound (type-level enforcement; no bypass path) |
| Concrete-integer filter on entry-safety store facts | Sound (prevents tautological candidates from variable-valued facts) |
| Whole-formula negation in entry-safety safety formula | Sound (prevents spurious type-bound atoms from contaminating candidates) |
| Latch node uses standard WP in inductiveness path | Conservative in failure mode (spurious rejection only) |
| `Call { HavocMemory }` transparent in standard WP | Relies on paired `HavocRegions` effect; missing `HavocRegions` → under-constrained backward state → more UNKNOWN, not false Verified |
| BMC `BugFound` | Sound — unrolled CFG is a concrete (non-approximating) prefix; SAT model is a real execution trace |
| BMC absence of bug | Not a proof — UNKNOWN for executions beyond the bound |

There is no false-Verified unsoundness in the current implementation.

## BMC as a Complementary Approach

When the invariant-based prover returns UNKNOWN (no invariant found), Bounded
Model Checking (`bmc.rs`) can find bugs by unrolling the loop k times and
checking one SAT query on the acyclic result. See `design_notes/BMC.md` for
the full design. The two approaches are complementary:

- **Invariant prover**: can verify safety; cannot find bugs when invariant synthesis fails.
- **BMC**: can find bugs in ≤ k iterations; cannot prove safety.
